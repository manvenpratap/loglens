Design Principles

Six Principles That Must Survive Every Phase

These are the constraints that differentiate LogLens — violate one and it becomes a worse version of an existing tool.

🔒
Privacy by Default
Log data never leaves the analyst's machine unless the analyst explicitly chooses a connector. Every new feature must pass a privacy review before shipping — including telemetry, update checks, and error reporting. Opt-in only, always.
⚡
Zero Mandatory Infrastructure
The single-file HTML build must remain fully functional forever. Collaboration and integration features are layered on top — never a prerequisite. A user behind a corporate proxy with no npm access must have the same core experience.
🧩
Metadata Over Magic
Every behaviour in the tool is driven by the rule configuration, not hidden heuristics. If the tool does something, the user must be able to trace it back to a specific rule or setting. No invisible AI black-boxes producing unexplained results.
📐
Format Agnosticism
LogLens must never hard-code a log format, framework, or vendor. Spring Boot, Tomcat, Node.js, Python, mainframe — all are first-class citizens through the rule engine. The Log4j importer is a convenience, not a prerequisite.
🚀
Time-to-First-Insight Under 5 Minutes
From opening LogLens to seeing the first meaningful parsed event must take less than 5 minutes for a new user on an unfamiliar log. Every UX decision — wizard, token builder, Log4j import, sample log — is evaluated against this constraint.
📦
Open Config Format
The rule JSON format must remain a plain, human-readable, Git-committable document that can be edited in any text editor without LogLens. Lock-in through an opaque binary format is a non-starter — it is the antithesis of the tool's value proposition.
